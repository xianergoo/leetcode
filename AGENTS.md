# 项目交流经验记录

> 每次对话中沉淀的可复用模式、教学偏好、约定。

## Git 操作规则

**所有Git操作都必须事先获得用户明确同意**，包括但不限于：

- 提交更改 (`git commit`)
- 推送到远程 (`git push`)
- 分支创建、合并 (`git branch`, `git merge`, `git rebase`)
- 任何破坏性操作 (`git reset`, `git checkout -f`)

**可自动执行的 Git 操作（无需确认）：**
- 只读操作 (`git status`, `git log`, `git diff`)

### 流程

1. 变更就绪后，展示 diff/staged 内容
2. 说明 commit message 建议
3. **等待用户确认**: "是否同意这次 commit/push？"
4. 只有在明确同意后才执行

## 交互节奏

1. **先对话探索，再写代码** — 不要一上来就丢代码，先讨论思路
2. **从暴力法出发** — 每道题先问最朴素的想法，然后分析为什么慢，再逐步优化
3. **理解优先** — 确认用户理解透彻后再写文件、做 commit
4. **对话即学习** — 思路讲解在对话中完成，文件只是最终沉淀

## Commit 规范

- 每道题一个 commit，message 格式：`<分类>: <题号>. <中文名> (<英文名>) - <难度>`
- 示例：`hash_table: 001. 两数之和 (Two Sum) - Easy`
- 每次 commit 同时更新 PROGRESS.md

## 题解文件格式

- 文件头部注释：题号、题目、难度、链接、描述、思路、复杂度
- 解法函数放文件内
- 测试单独 `_test.go` 文件
- `go test -v ./solutions/<category>/` 验证

## 已积累的思维技巧

1. **查找即哈希** — 当问题需要频繁判断「某个值是否存在/出现过」，立刻想到 map，O(n²) 变 O(n)。（来源：001. 两数之和）
2. **签名分组** — 需要把同类元素聚类时，给每个元素算一个「签名」（排序/编码），签名相同即同组，map 的 key 用签名。（来源：049. 字母异位词分组）
3. **模板+自主实现** — 探讨完思路后，给用户写好的模板（函数签名+注释+测试），让用户自己动手写实现体，比对代码丢过去效果好得多。
4. **只从起点开始** — 遍历序列类问题时，只从序列的「起点」（num-1 不存在）开始向后扩展，避免每个元素都重复计数。（来源：128. 最长连续序列）
