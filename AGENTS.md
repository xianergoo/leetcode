# 项目交流经验记录

> 每次对话中沉淀的可复用模式、教学偏好、约定。

## 交互节奏

1. **先对话探索，再写代码** — 不要一上来就丢代码，先讨论思路
2. **从暴力法出发** — 每道题先问最朴素的想法，然后分析为什么慢，再逐步优化
3. **理解优先** — 确认用户理解透彻后再写文件、做 commit
4. **对话即学习** — 思路讲解在对话中完成，文件只是最终沉淀

## Commit 规范

- 每道题一个 commit，message 格式：`<分类>: <题号>. <中文名> (<英文名>) - <难度>`
- 示例：`hash_table: 001. 两数之和 (Two Sum) - Easy`
- 每次 commit 同时更新 PROGRESS.md

## 题解文件格式

- 文件头部注释：题号、题目、难度、链接、描述、思路、复杂度
- 解法函数放文件内
- 测试单独 `_test.go` 文件
- `go test -v ./solutions/<category>/` 验证

## 已积累的思维技巧

1. **查找即哈希** — 当问题需要频繁判断「某个值是否存在/出现过」，立刻想到 map，O(n²) 变 O(n)。（来源：001. 两数之和）
